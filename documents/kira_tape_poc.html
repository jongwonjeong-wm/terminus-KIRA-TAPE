Terminus-KIRA + TAPE PoC Report

Terminal-Bench 2.0 · Jongwon Jeong · April 2026

TL;DR: We integrated TAPE on top of Terminus-KIRA. On a selected subset of 10 Terminal-Bench 2.0 tasks, TAPE improves solve rate from 4/10 to 8/10 with no regressions, while reducing average cost and execution time.

1. Introduction

We adapt two modules from TAPE [Jeong et al., 2026] — solver-integrated planning and state mismatch detection with replanning — as an outer wrapper on top of Terminus-KIRA [KRAFTON AI & Ludo Robotics, 2026] to complement its planning ability. TAPE decomposes LLM agent failures into sampling errors and planning errors, and proposes corresponding modules to address each. In Terminal-Bench tasks, models at the Haiku tier and above exhibit negligible sampling errors, and Terminus-KIRA already enforces structured outputs via native tool calling; we therefore focus solely on the planning-side modules. The resulting architecture is shown in Figure 1. Our implementation is available at https://github.com/jongwonjeong-wm/terminus-KIRA-TAPE.

Figure 1. Agent Architecture: Base Agent (Terminus-KIRA) + TAPE

Figure 1 shows the overall architecture. The left side contains the TAPE planning pipeline (purple) and the base agent (green), while the right side shows the tmux terminal where commands are executed. The two cards below summarize each component, and the execution flow follows.

Base Agent: Terminus-KIRA (green box in Figure 1)

Harness change from Terminus-2.

  • Native tool calling — replaces ICL JSON/XML parsing with structured tool use
  • image_read tool — new tool for visual analysis of files
  • Marker-based polling — detects command completion early, saves wait time
TAPE Modules (purple/yellow boxes in Figure 1)

Wraps Terminus-KIRA with planning and monitoring layers.

Planning

  • Planner — generates M candidate plans from task description
  • Simulator — predicts expected state for each subgoal
  • Graph Builder + ILP Solver — builds dependency graph, selects optimal path

Monitoring

  • State Mismatch Detection — compares terminal state vs. predicted state, outputs: ADVANCE / IN_PROGRESS / REPLAN

Execution Flow

The following describes the end-to-end execution flow, corresponding to the arrows in Figure 1.

  1. Task arrives → TAPE’s planning pipeline (purple box in Figure 1) generates M candidate plans, simulates expected states, builds a dependency graph, and the ILP solver selects the optimal path
  2. For each subgoal in the selected path:
    • TAPE dispatches the subgoal to Terminus-KIRA (green box), injecting subgoal context + tool_choice forcing into the prompt (first step only)
    • Terminus-KIRA calls LLM → gets commands → executes via tmux (right column) → reads terminal output
    • TAPE’s State Mismatch Detection (yellow box) compares terminal output against predicted state:
      • ADVANCE — subgoal complete, dispatch next subgoal
      • IN_PROGRESS — continue same subgoal, no forcing
      • REPLAN — state diverged, re-run full planning pipeline from current state
  3. All subgoals complete → task done

One Non-trivial Part: Double Confirmation (task_complete)

The base agent (Terminus-KIRA) has a built-in double confirmation for task_complete: the agent must call it twice consecutively to finish.

  • Base agent re-confirms — task completes normally
  • Base agent backs out (decides more work is needed) — TAPE triggers a REPLAN from current state

3. Overall Results

TAPE improves Terminus-KIRA across all key metrics: success rate doubles (4/10 → 8/10), average API cost drops 28%, and average execution time drops 11% — with zero regressions.

Summary:
  1. Task success rate doubles — 4/10 → 8/10 (+4 new solves, 0 regressions)
  2. Average API cost reduced 28% — $1.51 → $1.09/task, fewer wasted episodes on complex tasks
  3. Average execution time reduced 11% — 12.6m → 11.2m, subgoal-guided execution avoids aimless exploration

4. Detailed Sample-wise Analysis

Qualitative analysis of individual task runs reveals the following findings.

Findings:
(+) What TAPE solves:
  1. Subgoal planning before execution cuts over-verification and aimless exploration (100%, Partial groups)
  2. ILP solver path selection avoids failure-prone execution orders (Partial group — Appendix B)
  3. Replanning recovers from unexpected errors and converges on correct strategies (Partial, 0% groups)
(–) What TAPE introduces:
  1. Planning overhead adds ~2x cost on simple tasks (100% group)
What remains unsolved:
  1. Capability limits (model-extraction, sqlite-db-truncate) remain unresolved regardless of planning ability

(+) Mismatch detection cuts unnecessary episodes: TAPE’s subgoal planning before execution defines clear stopping criteria per step — the mismatch detection moves the agent to the next subgoal on completion, preventing over-verification loops. On kv-store-grpc (45 → 25 episodes), Terminus-KIRA spent episodes 26–44 on repetitive test cycles after a correct implementation; on portfolio-optimization (94 → 80), it hit repeated 60s timeout cycles without clear progression.

(–) TAPE planning + replanning adds cost on simple tasks: TAPE’s upfront planning (M=4 candidates → simulation → graph → ILP) and per-episode mismatch detection LLM calls increase token usage. Replans amplify this — fix-git had 1 replan (bash syntax error), portfolio-optimization had 2 — each re-running the full pipeline. Negligible on complex tasks, but visible here.

(+) Subgoal planning before execution prevents inefficient exploration: On code-from-image, Terminus-KIRA looped through 400+ verification steps after an already-correct solution ($2.99); TAPE finished in 12 calls ($0.08) because subgoal planning before execution gave each step clear completion criteria. Two replans further adapted the plan after permission errors and hook failures, completing in 44 calls — while Terminus-KIRA got stuck in a heredoc and timed out (12 calls).

(–) model-extraction fails for both — capability limit, not planning: Both agents extracted the matrix but achieved ~9.3% reconstruction error, short of the required accuracy. This is a Haiku capability limit on analytical ML attack strategy, not addressable by planning alone.

(+) Replanning converges on correct strategy for large-scale-text-editing: TAPE succeeded on a 0%-base-rate task through 11 replanning cycles that iteratively refined a Vim macro approach for bulk CSV transformation (157 calls, $1.22). Terminus-KIRA wrote a near-correct script but omitted a single execution line (:%normal! @c), then entered a false-completion loop — spending 100+ episodes echoing “Task Complete” without ever recognizing the bug (302 calls, $1.76). TAPE’s mismatch detection prevented this: each replan re-evaluated terminal state against predicted state, catching incomplete execution and re-triggering the planning pipeline.

(–) sqlite-db-truncate fails for both — domain knowledge gap: Both agents attempted raw binary parsing of the truncated SQLite file instead of using built-in recovery tools (sqlite3 .recover). TAPE scored 6/10 correct values (needed >6), Terminus-KIRA scored 0/10. TAPE’s planning overhead made it 7x more expensive ($0.65 vs $0.09) without improving the outcome — the bottleneck is domain knowledge (SQLite page format parsing), not task structure.

5. Conclusion & Next Steps

Findings from TAPE’s controlled experiments (Sokoban) transfer to real-world terminal tasks: solve rate 4/10 → 8/10, 0 regressions, 28% lower average cost. Planning overhead on simple tasks and base-model capability limits remain open.

Next steps:

  1. Statistical validation — scale to 5 runs on the 10 selected tasks to confirm these findings are statistically significant
  2. Full benchmark evaluation — expand to all 89 Terminal-Bench 2.0 tasks to measure aggregate score improvement
  3. Address remaining limitations — investigate stronger models (Sonnet) for capability-limited tasks (model-extraction, sqlite-db-truncate), and reduce planning overhead on simple tasks

Appendix B. ILP Path Selection: git-multibranch

The merged plan graph for git-multibranch (tape-plan-0) is the clearest example of solver-integrated planning adding value. Four candidate plans (9–18 steps) were merged into a 27-node, 38-edge graph. The ILP selected a 13-step path (total reward = 12.3) that made two non-trivial routing decisions.

Figure 2. Merged Plan Graph for git-multibranch (ILP-selected path highlighted)

Key insight: At node 7, the ILP chose the full E2E test path 7→10→11→8 (total reward=2.76, cost=5.6) over the greedy shortcut 7→8 (total reward=0.99, cost=0.1). A greedy policy would take the shortcut — lower cost and direct to goal. But the ILP maximizes total path reward: the longer route collects 2 goal nodes and nearly 3x the reward. The end-to-end test turned out to be essential — the verifier required actual deployment verification, not just “setup complete.”

In compile-compcert (tape-plan-4), a similar pattern emerged: a 21-node graph with ILP selecting a reward-3.0 path over alternatives scoring 2.0, routing around a high-risk “build from source” edge (r=0.65).
